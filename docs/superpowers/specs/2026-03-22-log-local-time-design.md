# 日志时间本地化设计

## 背景

用户反馈：
- 日志中的 ISO 时间（带 T 和 Z）"反人类"
- 需要手动转换成本地时间才能理解
- 之前的 spec 中"日志用 ISO 时间"的决策不符合实际使用需求

## 核心原则

| 时间类型 | 使用场景 | 格式 |
|---------|---------|------|
| **机器消费的时间** | 保存到文件、数据库、JSON API | ISO 8601 (UTC) |
| **人类阅读的时间** | CLI 输出、日志文本 | 本地时间 |

## 设计决策

### 保持 ISO 时间的部分
1. **存储层**：`.cadence/` 目录下的所有 JSON/YAML 文件中的时间戳
   - `nextRunAt`（任务状态）
   - `startedAt`、`finishedAt`（执行历史）
   - 理由：精确、无歧义、可排序、跨时区兼容

2. **JSON 输出**：`--json` 标志的输出
   - 理由：API 兼容性，机器消费

### 改用本地时间的部分
1. **CLI 输出**（已实现）
   - `cadence cron` 命令
   - `cadence status` 命令
   - 理由：给人看的

2. **所有文本日志**
   - `"Task scheduled"` 日志中的 `nextRun`
   - 任务执行日志中的时间
   - 其他所有人类阅读的日志输出
   - 理由：友好、直观、无需转换

## 修改文件清单

| 文件 | 修改内容 |
|------|----------|
| `src/core/scheduler/index.ts` | 日志中的 `nextRun` 使用 `formatLocalTime()` |
| `src/core/executor/index.ts` | 执行日志中的时间使用本地时间（如需要） |
| 其他日志输出位置 | 按需修改 |

## 不修改的部分

- 所有存储到磁盘的时间戳继续用 `toISOString()`
- JSON API 输出继续用 ISO 时间
- `formatLocalTime()` 和 `formatLocalTimeWithOffset()` 工具无需改动

## 理由总结

1. **实用主义**：开发者看日志时需要的是本地时间，不是 UTC
2. **一致性**：CLI 输出和日志都用本地时间，体验统一
3. **技术正确性**：存储层保持 ISO，确保数据精确可移植
